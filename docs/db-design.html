📐 개요 & 설계 원칙

앱 소개 Match-up은 테니스·풋살 동호인을 위한 대회 정보, 클럽 검색, AI 챗봇 앱입니다. 사용자 한 명이 여러 종목에 등록할 수 있으며, 종목별로 별도의 등급·협회를 관리합니다.
설계 원칙
  • 모든 테이블 RLS ON — 서버가 권한의 진실
  • sport 컬럼으로 종목 구분 (테이블명에 종목명 없음)
  • UUID v7 (시간순 정렬 가능) — crawl_sources 부터 적용
  • pgvector 768차원 — Gemini embedding-001 기준
  • No TypeScript any, No Dart dynamic
확정된 설계 결정
  • 카카오 로그인 — 이메일 필수 (NOT NULL 유지)
  • 한 사용자 N개 종목 가능 (user_sports)
  • 한 사용자 N개 클럽 가입 가능 (club_memberships 예정)
  • 종목별 활동 지역 별도 관리 (user_sports.region_code 예정)
  • 관리자 승인 없이 클럽 등록 불가 (현재 정책 유지)

변경: user_sports에 region_code 추가

종목별 활동 지역을 별도로 관리. 테니스는 광주, 풋살은 서울처럼 종목마다 다른 지역 가능.

ALTER TABLE user_sports ADD COLUMN region_code text REFERENCES regions(code);

논의 1 — user_role enum 확장성

미결정

현재 상태

현재 user_role enum은 user | admin 2개만 존재합니다.

문제

PostgreSQL enum은 값 추가(ADD VALUE)는 가능하지만 삭제·이름 변경·순서 변경이 불가합니다. 나중에 역할이 세분화되면(예: 콘텐츠 관리자, 클럽 관리자 등) enum 자체를 교체해야 하는 작업이 발생합니다.

✅ 옵션 A — enum에 미리 추가

지금 staff를 미리 추가해두고 향후 사용. enum은 그대로 유지.

user | staff | admin

장점: 간단, 현재 코드 변경 최소화
단점: 역할이 더 세분화되면 여전히 한계

🔵 옵션 B — permissions jsonb 추가

users에 permissions jsonb 컬럼 추가. 역할별 권한을 유연하게 표현.

예: {"can_approve": true, "can_crawl": true}

장점: 무한 확장 가능, 세밀한 권한 제어
단점: RLS 정책 복잡해짐, 타입 안전성 약해짐

논의 포인트 어드민이 앞으로 얼마나 세분화될 예정인가요? 현재 1~2명의 어드민이라면 A안으로 충분합니다. 여러 팀원이 각각 다른 권한을 가져야 한다면 B안을 검토하세요.

논의 5 — 컬럼 수와 테이블 분리 수준 (참고)

팀 확인 권장

tournaments 테이블이 30개 컬럼으로 많아 보일 수 있지만 PostgreSQL에서 컬럼 수는 성능 문제가 없습니다. 필요한 컬럼만 SELECT하면 되고, 인덱스 구조가 적절히 잡혀 있습니다.

우려 사항실제 영향판단
tournaments 30개 컬럼 SELECT 성능에 영향 없음. 인덱스 기반 조회. 문제없음
user_sports / user_tennis_orgs 분리 JOIN 1회 추가. 규모에서 무시 가능. 문제없음
region text + region_code FK 중복 데이터 불일치 버그 가능성 논의 2 참조
user_role enum 2개값 세분화 시 ALTER TYPE 필요 논의 1 참조
결론 현재 설계는 MVP 수준에서 충분합니다. 지금 당장 뜯어고칠 필요는 없습니다. 논의 1(역할 확장)과 논의 2(지역 중복)만 방향을 정해두면 나머지는 실제로 불편해질 때 리팩터링해도 늦지 않습니다.
