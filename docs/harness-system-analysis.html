1. 현재 진단

코드는 빌드 가능한 상태지만, 제품 규칙이 자동으로 깨지지 않는다는 보장은 아직 약합니다.

확인된 기술 스택

Flutter / Dart Riverpod GoRouter Supabase Edge Functions Deno / TypeScript Postgres + RLS pgvector + Gemini FCM

현재 검증 상태

flutter analyze 통과 flutter test 통과 deno check 통과 deno lint 통과 도메인 테스트 부족 SQL/RPC 테스트 부족 보안 하네스 부족
좋은 점

핵심 등급 필터링이 DB RPC에 있고, RLS가 주요 테이블에 적용되어 있습니다.

아쉬운 점

region_code, host_orgs 필터는 아직 RPC가 아니라 Edge Function 후처리입니다.

우선 보완

입력 길이 제한, RAG prompt injection 방어, cron secret, rate limit 하네스가 필요합니다.

2. 하네스 시스템 4계층

모든 규칙을 한 곳에서 처리하려 하지 말고, 책임별로 나누어 작고 확실한 게이트를 만듭니다.

1

도메인 규칙 하네스

Match-up의 핵심 가치인 출전 가능 대회 자동 필터링을 검증합니다.

  • sport/grade/org/region enum 일치
  • 종목 교차 매칭 방지
  • 권역·협회 필터 pagination 전 처리
  • published/draft 노출 규칙
2

보안/운영 하네스

비용 폭탄, 데이터 오염, 권한 우회를 막습니다.

  • cron secret
  • chat/semantic-search rate limit
  • RAG prompt injection 방어
  • secret scan
3

코드 품질 하네스

언어별 규칙을 PR마다 자동 확인합니다.

  • flutter analyze/test
  • deno fmt/lint/check/test
  • any, dynamic, ! 예외 감시
  • CI merge gate
4

제품 기능 하네스

사용자가 체감하는 기능의 품질을 검증합니다.

  • 대회 검색/제보/승인
  • 챗봇/RAG
  • 알림 dedup
  • 스피드건 계산 정확도

3. 규칙 카탈로그

아래 규칙은 “문서에 적기”가 아니라 “테스트나 스크립트로 실패시키기”가 목표입니다.

우선순위 규칙 강제 위치 권장 하네스
P0 서버/DB가 최종 판정자다.
Flutter는 표시와 입력 보조만 담당하고, eligibility·권한·검색 판정을 신뢰하지 않는다.		 DB RPC, Edge Function SQL smoke test, RPC test
P0 RAG 컨텍스트는 명령이 아니라 데이터다.
대회 설명·룰북·웹 결과 안의 지시문은 무시되어야 한다.		 chat/index.ts Deno unit test, prompt safety snapshot
P0 비용 발생 함수는 공개 엔드포인트처럼 두지 않는다.
cron, chat, semantic-search는 secret 또는 rate limit이 필요하다.		 Edge Functions, DB security harness, 429 test
P1 sport/grade/org/region enum은 Dart/TS/SQL 간 불일치하면 실패한다. grade_labels.dart, enums.ts, migrations scripts/harness/check_enums.py
P1 검색 필터는 pagination 전에 적용한다.
region_code, host_orgs 후처리 필터는 임시로만 허용한다.		 tournaments_for_user_v2 SQL fixture test
P1 사용자 입력은 저장 전에 검증한다.
길이, enum, URL scheme, 배열, 날짜 형식을 모두 확인한다.		 tournaments-submit, chat, semantic-search validation.ts + Deno test
P1 새 테이블은 RLS enabled + 정책 없이는 병합하지 않는다. Supabase migrations SQL metadata check
P2 TypeScript any는 금지한다.
외부 JSON은 unknown으로 받고 type guard로 좁힌다.		 Supabase functions Deno lint + static grep
P2 Dart dynamic은 JSON boundary에서만 제한적으로 허용한다.
모델 내부 필드는 typed model로 변환한다.		 Flutter models/services static rule script + model tests
P2 스피드건은 입력 품질이 낮으면 결과를 내지 않거나 신뢰도 낮음으로 표시한다. Speed Gun services/screens speed_calculator_test, fixture test
P3 문서 규칙은 자동화된 하네스에 연결한다.
문서만 있는 규칙은 시간이 지나면 반드시 어긋난다.		 docs/rules/*.md PR checklist + CI link

6. 로컬 실행 명령

하네스는 개발자가 PR 올리기 전에 로컬에서 같은 검증을 돌릴 수 있어야 합니다.

권장 run_all.sh 동작 
#!/usr/bin/env bash
set -euo pipefail

ROOT="$(cd "$(dirname "$0")/../.." && pwd)"

cd "$ROOT/app"
flutter pub get
flutter analyze
flutter test

cd "$ROOT/supabase/functions"
deno fmt --check */index.ts _shared/*.ts
deno lint --config deno.json */index.ts _shared/*.ts
deno check --config deno.json */index.ts _shared/*.ts
deno test --config deno.json --allow-env --allow-read

cd "$ROOT"
python3 scripts/harness/check_enums.py
python3 scripts/harness/check_static_rules.py

현재 확인된 통과 명령

flutter analyze flutter test deno check deno lint

현재는 이 명령들이 통과합니다. 다음 단계는 여기에 도메인/보안/스피드건 테스트를 추가해서 “기능이 망가지지 않았다”까지 확인하는 것입니다.

7. 스피드건 전용 규칙

스피드건은 결과 숫자가 사용자 신뢰와 직결됩니다. 틀린 값을 예쁘게 보여주는 것보다, 입력 품질이 낮을 때 경고하는 것이 중요합니다.

입력 품질

권장: 120fps 미만은 경고, 60fps 미만은 분석 차단 또는 “신뢰 낮음” 표시.

fps gate 영상 길이 제한 해상도 확인

Calibration

코트 4점은 개수뿐 아니라 비퇴화 좌표인지 확인해야 합니다. assert만으로는 부족합니다.

4점 필수 중복 좌표 금지 면적 기준

결과 sanity check

예: 0~300km/h 범위를 벗어나면 측정 실패 또는 낮은 신뢰도로 처리합니다.

peak range avg range detected frames
현재 모델 에셋 상태

app/assets/models/yolov8n.tflite는 아직 없고, 현재 BallDetectorService는 모델이 없으면 mock 모드로 동작합니다. 개발 중에는 좋지만, production build에서는 mock 사용을 막는 하네스가 필요합니다.

8. GitHub Actions 예시

처음에는 Flutter, Deno, static-rules 세 job으로 나누는 구성이 가장 단순합니다.

.github/workflows/harness.yml 
name: harness

on:
  pull_request:
  push:
    branches: [main]

jobs:
  flutter:
    runs-on: macos-latest
    defaults:
      run:
        working-directory: app
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          channel: stable
      - run: flutter pub get
      - run: flutter analyze
      - run: flutter test

  deno:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: supabase/functions
    steps:
      - uses: actions/checkout@v4
      - uses: denoland/setup-deno@v2
      - run: deno fmt --check */index.ts _shared/*.ts
      - run: deno lint --config deno.json */index.ts _shared/*.ts
      - run: deno check --config deno.json */index.ts _shared/*.ts
      - run: deno test --config deno.json --allow-env --allow-read

  static-rules:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: python3 scripts/harness/check_enums.py
      - run: python3 scripts/harness/check_static_rules.py

9. Definition of Done

하네스가 적용된 뒤에는 PR 완료 기준을 명확히 잡아야 합니다.

모든 PR 공통
  • scripts/harness/run_all.sh 통과
  • 새 규칙 추가 시 문서와 테스트가 함께 추가됨
  • 새 테이블은 RLS enabled + 정책 포함
  • 새 사용자 입력은 validation test 포함
  • 불가피한 as, !, JSON dynamic은 사유가 명확함
Edge Function PR
  • deno fmt --check, deno lint, deno check 통과
  • req.json() 결과는 unknown 또는 validation 함수 통과 후 사용
  • 비용 발생 API 호출은 rate limit 또는 인증 방어 포함
  • RAG에 들어가는 외부 데이터는 데이터 블록으로 격리
Flutter PR
  • flutter analyze, flutter test 통과
  • 도메인 모델 파싱 테스트 추가
  • 사용자에게 노출되는 오류 메시지가 raw exception만은 아님
  • web/mobile 플랫폼 분기 명확
스피드건 PR
  • 계산 로직은 fixture 기반 단위 테스트 포함
  • 입력 품질이 낮을 때 경고 또는 차단
  • prod build에서 mock detector 사용 금지
  • 속도 결과 sanity check 포함
